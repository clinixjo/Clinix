# خطة التنفيذ الكاملة (Master Roadmap) — SaaS عيادات التجميل

> **الغرض:** الخريطة الكاملة من الصفر حتى الإطلاق. كل milestone بهدفه، ومهامه، ومخرجاته، ومعيار اعتباره منتهيًا (Definition of Done). تُقرأ مع وثيقة التخطيط ونظام التصميم.
> **الفريق:** مهندس رئيسي (أنت + Claude Code) بأدوار متعددة: Web Developer، UI/UX، Architect، Graphic Designer، CMO.
> **مدة الوصول للإطلاق (تقديري):** ١٣–١٨ أسبوعًا لفرد واحد مع Claude Code.
> **آخر تحديث:** تموز ٢٠٢٦

---

## ١. كيف يتوزّع "الفريق" على العمل

| الدور | مسؤوليته في المشروع |
|---|---|
| **Architect** | معمارية تعدد المستأجرين، نموذج البيانات، RLS، الأمان والامتثال، قرارات البنية |

| **Graphic Designer** | الهوية البصرية، الشعار، نظام التصميم، أصول التسويق البصرية |
| **CMO** | التموضع، التسعير، الموقع التعريفي، استقطاب العيادات، مواد البيع، الإطلاق |

> **مبدأ العمل مع Claude Code:** milestone واحد في كل مرة. أعطِ Claude Code وثيقة التخطيط + نظام التصميم + هذا الملف كسياق، ثم وجّهه لـ milestone محدد. لا تنتقل للتالي قبل تحقيق معيار الإنجاز.

## ٣. المراحل بالتفصيل

### 🏗️ M0 — الأساس والإعداد
**الهدف:** بنية تحتية صلبة وآمنة قبل أي ميزة. *أهم مرحلة على الإطلاق — أي خطأ هنا مكلف لاحقًا.*
**القائد:** Architect

**المهام:**
- تهيئة المشروع: Next.js (App Router) + TypeScript + Tailwind CSS + shadcn/ui.
- إعداد مشروع Supabase (منطقة قريبة مناسبة) + متغيّرات البيئة.
- تصميم مخطط قاعدة البيانات الكامل (كل الجداول من وثيقة التخطيط).
- **تطبيق تعدد المستأجرين:** عمود `clinic_id` على كل جدول + سياسات **Row Level Security** تمنع أي تسريب بين العيادات.
- إعداد المصادقة (Supabase Auth) + الأدوار (owner/admin/receptionist/practitioner).
- إعداد الترجمة (`next-intl`) بلغتين + تبديل `dir` (RTL/LTR).
- تحويل design tokens إلى متغيّرات CSS + `tailwind.config`.
- بناء المكوّنات المشتركة الأساسية: Button, Card, Input, StatusBadge, MetricCard, Avatar, ListRow, AppShell (تخطيط + تنقّل).
- إعداد النشر (Vercel) + خط CI بسيط.

**المخرجات:** مشروع يعمل، تسجيل دخول، هيكل تنقّل فارغ، قاعدة بيانات مؤمّنة.

**✅ معيار الإنجاز:**
- مستخدم من العيادة (أ) **لا يستطيع بأي طريقة** رؤية أو تعديل بيانات العيادة (ب) — مُختبَر بكتابة اختبار عزل صريح.
- تبديل اللغة يقلب الاتجاه والنصوص بالكامل.
- الأدوار تعمل: كل دور يرى ما يخصّه.

---

### 👥 M1 — العملاء (CRM)
**الهدف:** قاعدة عملاء منظمة — حل المشكلة الأولى.
**القائد:** Web Developer

**المهام:**
- CRUD كامل للعميل: اسم، عمر/تاريخ ميلاد، جنس، واتساب، هاتف، بريد، **مصدر العميل**، ملاحظات.
- **الملاحظات الطبية** بوصول محدود (طبيب/مالك فقط) على مستوى RLS.
- **نموذج الموافقة الرقمي** عند إنشاء عميل جديد (مع تسجيل وقت الموافقة).
- ملف العميل: عرض بياناته + سِجل زياراته وخدماته (يمتلئ لاحقًا مع M3/M4).
- بحث وفلترة (بالاسم/الهاتف).

**المخرجات:** وحدة عملاء كاملة قابلة للاستخدام.

**✅ معيار الإنجاز:** إضافة/تعديل/بحث عن عميل يعمل، الموافقة مسجّلة، الملاحظات الطبية محجوبة عن الاستقبال.

---

### 💉 M2 — الخدمات والباقات
**الهدف:** كتالوج خدمات ونظام باقات — أساس للمبيعات وإعادة الاستهداف.
**القائد:** Web Developer

**المهام:**
- CRUD للخدمات: اسم، فئة، سعر، مدة، **فترة المتابعة الموصى بها** (`followup_interval_days`).
- دعم الباقات: عدد جلسات + تتبّع المتبقي لكل عميل (`patient_packages`).
- ربط الخدمة بالفئات (ليزر/حقن/عناية...).

**✅ معيار الإنجاز:** يمكن تعريف خدمات مفردة وباقات، وعدّاد جلسات الباقة يعمل.

---

### 📅 M3 — المواعيد (التقويم)
**الهدف:** حجز منظّم بدل الإكسل — حل المشكلة الثانية.
**القائد:** Web Developer + UI/UX

**المهام:**
- تقويم يومي/أسبوعي، عرض مواعيد اليوم.
- تدفّق حجز سريع (≤ ٣ نقرات): عميل + خدمة + طبيب + وقت.
- حالات الموعد: مجدول/مؤكد/مكتمل/غياب/ملغى (بالألوان الدلالية).
- منع تعارض المواعيد في نفس الوقت للطبيب نفسه.
- عند "مكتمل" → يُنشأ سجل زيارة تلقائيًا في ملف العميل.
- قراءة إعداد `practitioner_can_edit` لتحديد صلاحية الطبيب حسب العيادة.

**✅ معيار الإنجاز:** حجز وتعديل موعد يعمل، لا تعارض، تحويل لمكتمل يسجّل زيارة، الاختبار في الاتجاهين.

---

### 💰 M4 — المبيعات والفواتير
**الهدف:** تتبّع ما يُباع فعليًا — حل المشكلة الثالثة (الجزء الأول).
**القائد:** Web Developer

**المهام:**
- تسجيل المبيعات وبنودها (`sales` + `sale_items`): الخدمة، السعر، الكمية، **الطبيب المنفّذ** (تمهيدًا للعمولات).
- طريقة الدفع، حالة الفاتورة.
- الربط بالزيارة/العميل، وخصم جلسة من الباقة عند البيع.

**✅ معيار الإنجاز:** تسجيل بيع يعمل ويظهر في ملف العميل وفي بيانات التقارير.

---

### 🔁 M5 — إعادة الاستهداف (المتابعات) — الميزة التنافسية
**الهدف:** تحويل قاعدة العملاء إلى مكينة تسويق — حل المشكلة الأولى (الجزء الأقوى).
**القائد:** Architect + Web Developer

**المهام:**
- منطق حساب **تاريخ المتابعة المتوقّع** = آخر خدمة + فترة المتابعة (أو الجلسة القادمة في الباقة).
- شاشة/لوحة **"متابعات مستحقة"** مع أسباب واضحة (تجديد فيلر، جلسة باقة، لم يزُر منذ...).
- **إرسال واتساب يدوي:** زر يولّد نص رسالة مخصّص (قوالب ar/en) ويفتح `wa.me`.
- **سجل التواصل** (`message_log`): تسجيل من تم التواصل معه ومتى — حتى في الوضع اليدوي.

**✅ معيار الإنجاز:** قائمة المتابعات تظهر بدقة، الضغط يولّد رسالة صحيحة ويُسجّل في السجل.

---

### 📊 M6 — التقارير ولوحة التحكم
**الهدف:** رؤية تطوّر التسويق والأداء.
**القائد:** Web Developer + UI/UX

**المهام:**
- تقارير: الإيراد حسب الخدمة/الشهر/الطبيب، أكثر الخدمات مبيعًا.
- **نسبة الغياب** (no-show) و**نسبة رجوع العملاء** (retention) و**مصادر العملاء**.
- لوحة التحكم الرئيسية (كما في التصوّر): بطاقات مؤشرات + مواعيد اليوم + المتابعات المستحقة.
- رسوم بيانية بسيطة (Recharts).

**✅ معيار الإنجاز:** الأرقام دقيقة ومطابقة للبيانات الفعلية، لوحة التحكم مكتملة.

---

### ⚙️ M7 — الإعدادات وتهيئة العيادة
**الهدف:** جعل النظام قابلًا للتشغيل لكل عيادة باستقلالية.
**القائد:** Web Developer

**المهام:**
- إعدادات العيادة: الاسم، الشعار، بيانات التواصل، **تبديل نمط `practitioner_can_edit`**.
- إدارة المستخدمين والأدوار (دعوة/تعطيل موظف).
- حقل حالة الاشتراك (`subscription_status`) — أساس للفوترة لاحقًا.
- تدفّق تهيئة عيادة جديدة (onboarding) من الصفر.

**✅ معيار الإنجاز:** يمكن إنشاء عيادة جديدة وتهيئتها وإدارة مستخدميها بالكامل.

---

### 🔒 M8 — التدقيق والأمان والامتثال والـ QA
**الهدف:** جاهزية للإنتاج ببيانات طبية حساسة.
**القائد:** Architect

**المهام:**
- **سجل تدقيق (Audit Log)** للعمليات الحساسة.
- **حقوق صاحب البيانات:** تصدير/حذف بيانات عميل بعينه (متطلب قانوني).
- مراجعة نص الموافقة (يُفضّل مراجعة محامٍ) + بند المعالجة خارج الأردن.
- **خطة الاستجابة للاختراق** (التزام الإخطار في القانون الأردني).
- تشفير، مراجعة صلاحيات، اختبار عزل المستأجرين مرة أخرى تحت الضغط.
- **QA شامل:** كل شاشة في RTL و LTR، على موبايل/تابلت/كمبيوتر، وإمكانية الوصول.
- مراجعة الأداء (سرعة التحميل، الاستعلامات).

**✅ معيار الإنجاز:** لا ثغرة عزل، حقوق البيانات تعمل، كل الشاشات تجتاز QA في الاتجاهين والأجهزة.

---

### 🧪 M9 — التجربة مع عيادات رائدة (Beta)
**الهدف:** التحقق من قيمة النظام في الواقع قبل الإطلاق العام.
**القائد:** الكل (بقيادة CMO للعلاقة)

**المهام:**
- اختيار ٢–٣ عيادات رائدة (من "أقوى عيادات الأردن" كما هو الهدف).
- تهيئتها ومرافقتها في أول أسبوع.
- جمع الملاحظات المنظّم (ما الذي يربكهم؟ ما الذي يوفّر وقتهم؟).
- إصلاح أهم المشاكل + تحسينات سريعة.
- **جمع شهادات/قصص نجاح** (مادة تسويقية ثمينة للإطلاق).

**✅ معيار الإنجاز:** عيادة واحدة على الأقل تستخدمه يوميًا وتؤكد أنه يوفّر وقتها/يزيد عودة عملائها.

---

### 🚀 M10 — الإطلاق
**الهدف:** فتح الباب للعيادات ككل + بدء التحصيل.
**القائد:** CMO

**المهام:**
- تثبيت **نموذج التسعير** (شهري ثابت؟ حسب عدد المستخدمين؟).
- **الفوترة:** يمكن البدء يدويًا (فاتورة شهرية) قبل أتمتة الدفع.
- **الموقع التعريفي (landing page)** بالعربية أساسًا + الإنجليزية.
- حملة إطلاق + مواد بيع + عرض توضيحي (demo).
- عملية بيع وتهيئة للعيادات الجديدة.

**✅ معيار الإنجاز:** عيادات جديدة تستطيع التسجيل/الاشتراك والبدء دون تدخّل تقني مكثّف.

## ٤. مسار التسويق (CMO) — يعمل بالتوازي من M6

لا ننتظر انتهاء البرمجة لنبدأ التسويق. يبدأ هذا المسار مبكرًا:

- **التموضع والرسالة:** "نظام يحوّل قاعدة عملائك إلى عملاء عائدين — وداعًا للإكسل والمواعيد المنسية." التركيز على العائد (رجوع العملاء + توفير الوقت)، لا على المزايا التقنية.
- **الجمهور:** أصحاب/مديرو عيادات التجميل في الأردن.
- **قنوات الاستقطاب:** تواصل مباشر مع العيادات، إنستغرام (المجال بصري)، توصيات من العيادات الرائدة.
- **المواد:** موقع تعريفي، عرض توضيحي قصير، دراسة حالة من مرحلة Beta، عرض أسعار واضح.
- **الهوية البصرية (Graphic Designer):** شعار، ألوان الهوية (متسقة مع نظام التصميم)، قوالب سوشال ميديا.
- **حاجز الثقة:** بما أن البيانات طبية، أبرِز الأمان والخصوصية كنقطة بيع ("بياناتك محمية ومعزولة").

- [ ] الموقع التعريفي منشور.
- [ ] عيادة Beta واحدة على الأقل راضية وتشهد.

## ٦. ما بعد الإطلاق (المرحلة ٢ فما فوق)

بترتيب أولوية مقترح بناءً على القيمة:
1. **WhatsApp API الأوتوماتيكي** (رسائل مجدولة تلقائية) — أكبر قفزة قيمة.
2. **صور قبل/بعد + توثيق الموافقة الرقمية.**
3. **عمولات الأطباء الآلية.**
4. **المخزون** (فيلر/مستهلكات).
5. **الحجز الإلكتروني للعملاء** من موقع العيادة.
6. **الدفع الإلكتروني + الفوترة الآلية للاشتراكات.**
7. **الاستضافة داخل الأردن** (كميزة بيع + امتثال أقوى).

## ٧. المخاطر وكيف نتعامل معها

| الخطر | التخفيف |
|---|---|
| تسريب بيانات بين العيادات | RLS + اختبار عزل صريح في M0 و M8 |
| بطء التبنّي من العيادات | Beta مبكرة + شهادات + تهيئة مرافقة |
| تعقيد الواتساب الرسمي وتكلفته | نبدأ يدوي (`wa.me`)، نرقّي للـ API بعد إثبات الحاجة |
| مخاطر الامتثال القانوني | موافقة صريحة + مراجعة محامٍ + حقوق البيانات + سجل تدقيق |
| توسّع النطاق (scope creep) | التزام صارم بنطاق MVP، وتأجيل كل ما عداه للمرحلة ٢ |
| اعتماد فرد واحد | توثيق كل شيء (هذه الملفات) + commits صغيرة متكررة |

## ٨. الخطوة التالية الفورية

ابدأ **M0**. أول قطعة ملموسة هي **مخطط قاعدة البيانات (SQL) + سياسات RLS**، لأنها تحدّد كل ما بعدها. بعدها تهيئة المشروع والمكوّنات المشتركة.

**التسلسل المقترح للجلسة الأولى مع Claude Code:**
1. أعطه: وثيقة التخطيط + نظام التصميم + هذا الملف.
2. ابدأ بـ: "نفّذ M0 — ابدأ بمخطط قاعدة البيانات وسياسات RLS، ثم اكتب اختبار عزل المستأجرين."
3. لا تنتقل لـ M1 قبل تحقيق معيار إنجاز M0.
