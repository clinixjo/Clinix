# وثيقة تخطيط — نظام SaaS لإدارة عيادات التجميل

> **الحالة:** مرحلة التخطيط — هذه الوثيقة هي المرجع الأساسي الذي سنبني منه على Claude Code.
> **آخر تحديث:** تموز ٢٠٢٦

---

## ١. الرؤية والملخص التنفيذي

نبني منتج **SaaS متعدد المستأجرين (multi-tenant)** يُباع لعيادات التجميل في الأردن كاشتراك. النظام يحل ثلاث مشاكل أساسية يواجهها أصحاب العيادات اليوم:

1. **لا يوجد قاعدة عملاء منظمة** → لا يمكنهم إعادة استهداف العملاء أو متابعتهم.
2. **حجز المواعيد يدوي أو على إكسل** → فوضى، تعارض مواعيد، وغيابات (no-show).
3. **لا يوجد تتبع للمبيعات والخدمات** → لا يعرفون ما الذي يُباع فعليًا ولا كيف يطوّرون تسويقهم.

**الميزة التنافسية الأساسية** التي تميّزنا عن أي CRM عام: النظام مصمّم حول **دورة حياة العلاج التجميلي** (الفيلر يتجدد كل فترة، الليزر باقة جلسات، البوتوكس دوري) — بحيث يتحوّل النظام إلى **مكينة إعادة استهداف ذكية** تعرف متى موعد الجلسة القادمة لكل عميل، بدل الرسائل الجماعية العشوائية.

## ٢. القرارات المثبّتة (خلاصة نقاشنا)

| القرار | الاختيار |

## ٣. نطاق الـ MVP — ماذا نبني الآن وماذا نؤجّل

**قاعدة القرار:** كل ميزة خارج حل المشاكل الثلاث الأساسية تُؤجَّل. الهدف إطلاق نسخة قابلة للاستخدام بسرعة، ثم التوسّع بناءً على ملاحظات العيادات الحقيقية.

### داخل الـ MVP (المرحلة ١)
- **إدارة المستأجرين:** كل عيادة لها بياناتها المعزولة تمامًا.
- **إدارة المستخدمين والأدوار:** مالك، مدير، استقبال، طبيب/ممارس.
- **قاعدة العملاء (CRM):** ملف كامل لكل عميل + سجل زياراته.
- **كتالوج الخدمات:** الخدمات المفردة + الباقات (مع عدّاد الجلسات).
- **المواعيد:** تقويم، حالات الموعد (مجدول/مؤكد/مكتمل/غياب/ملغى)، تذكيرات.
- **تسجيل المبيعات:** ما الذي بيع، بأي سعر، بواسطة أي طبيب.
- **إعادة الاستهداف اليدوية:** قائمة "عملاء حان موعد متابعتهم" + توليد رسالة واتساب جاهزة بضغطة زر.
- **التقارير الأساسية:** الإيراد حسب الخدمة/الشهر/الطبيب، نسبة الغياب، نسبة رجوع العملاء.
- **دعم لغتين + RTL/LTR كامل.**

### خارج الـ MVP (مراحل لاحقة)
- WhatsApp API الأوتوماتيكي (رسائل تلقائية مجدولة).
- صور قبل/بعد + توثيق الموافقة الرقمية.
- المخزون (تتبع مخزون الفيلر/المستهلكات).
- عمولات الأطباء الآلية.
- الدفع الإلكتروني وربط الفواتير.
- الفوترة الآلية لاشتراكات العيادات نفسها.
- الحجز الإلكتروني للعملاء (booking من موقع العيادة).

## ٤. الأدوار والصلاحيات (Roles & Permissions)

| الدور | الصلاحيات |
|---|---|
| **المالك (Owner)** | كل شيء + إعدادات العيادة + إدارة المستخدمين + كل التقارير المالية |
| **المدير (Admin)** | إدارة يومية كاملة + معظم التقارير، بدون إعدادات الاشتراك الحساسة |
| **الاستقبال (Receptionist)** | إضافة/تعديل عملاء، حجز المواعيد، تسجيل المبيعات، إرسال رسائل الواتساب |
| **الطبيب/الممارس (Practitioner)** | عرض مواعيده وعملائه + إضافة ملاحظات علاجية. **صلاحية إدخال الخدمات/المبيعات قابلة للضبط حسب العيادة** (انظر أدناه) |

> **صلاحية الطبيب قابلة للضبط:** لكل عيادة إعداد (`practitioner_can_edit`) يحدّد نمط عملها:
> - **نمط "الطبيب يُدخل":** الطبيب يسجّل الخدمات والملاحظات بنفسه.
> - **نمط "متابعة فقط":** الاستقبال يُدخل كل شيء، والطبيب يعرض ويضيف ملاحظات فقط.
>
> هذا يعني أن نظام الصلاحيات ليس ثابتًا بل يقرأ إعداد العيادة. تعقيد بسيط لكن يجب بناؤه من البداية لتجنّب إعادة الهيكلة لاحقًا.

> **ملاحظة أمنية مهمة:** الملاحظات الطبية الحساسة يجب أن تكون محدودة الوصول (الطبيب والمالك)، وليست مرئية بالضرورة لكل موظف استقبال. سنطبّق هذا على مستوى قاعدة البيانات.

## ٥. الوحدات الأساسية (Modules) بالتفصيل

### ٥.١ وحدة العملاء (CRM)
ملف كل عميل يحتوي: الاسم، العمر/تاريخ الميلاد، الجنس، رقم الواتساب/الهاتف، البريد (اختياري)، **مصدر العميل** (كيف عرف العيادة — مهم جدًا للتسويق)، ملاحظات عامة، وملاحظات طبية (وصول محدود)، وسِجل كامل بكل الزيارات والخدمات والمبيعات.

### ٥.٢ وحدة الخدمات والباقات
كل خدمة لها: اسم، فئة (ليزر/حقن/عناية بشرة...)، سعر، مدة تقريبية، و**فترة المتابعة الموصى بها** (مثلاً: فيلر → متابعة بعد ٩ أشهر). الباقات: عدد الجلسات الكلي + تتبع المتبقي لكل عميل اشترى الباقة.

### ٥.٣ وحدة المواعيد
تقويم يومي/أسبوعي، ربط الموعد بعميل + خدمة + طبيب، منع التعارض في الوقت، وحالات واضحة للموعد. عند تحويل الموعد إلى "مكتمل" يُسجَّل تلقائيًا كزيارة/خدمة في ملف العميل.

### ٥.٤ وحدة إعادة الاستهداف (القلب النابض)
النظام يحسب لكل عميل **تاريخ المتابعة المتوقّع** = تاريخ آخر خدمة + فترة المتابعة الموصى بها (أو الجلسة القادمة في الباقة). ينتج عن ذلك قائمة يومية: **"عملاء حان وقت التواصل معهم"** — يضغط الموظف زرًّا فيتولّد نص رسالة واتساب مخصّص ويُفتح `wa.me` جاهزًا للإرسال. كل رسالة تُسجَّل في سجل التواصل (حتى في الوضع اليدوي) لنعرف من تم التواصل معه ومتى.

### ٥.٥ وحدة المبيعات والفواتير
تسجيل ما بيع في كل زيارة، بأي سعر، وبواسطة أي طبيب (تمهيدًا للعمولات لاحقًا)، وطريقة الدفع. هذه البيانات تُغذّي التقارير.

### ٥.٦ وحدة التقارير
- الإيراد حسب الخدمة / الشهر / الطبيب.
- أكثر الخدمات مبيعًا.
- نسبة الغياب (no-show rate).
- نسبة رجوع العملاء (retention) — مؤشر صحة العيادة الأهم.
- مصادر العملاء (من أين يأتون) — لتوجيه ميزانية التسويق.

## ٦. نموذج البيانات (Data Model)

الكيانات الأساسية والعلاقات بينها. **كل جدول (عدا `clinics`) يحتوي على `clinic_id`** لعزل المستأجرين.

- **clinics** (المستأجر): `id`, `name`, `slug`, `settings` (يشمل `practitioner_can_edit`), `subscription_status`, `created_at`
- **users**: `id`, `clinic_id`, `role`, `name`, `email`, `phone`, `is_active`
- **patients**: `id`, `clinic_id`, `name`, `dob`/`age`, `gender`, `whatsapp`, `phone`, `email`, `source`, `notes`, `medical_notes`, `created_at`
- **services**: `id`, `clinic_id`, `name`, `category`, `price`, `duration_min`, `is_package`, `sessions_count`, `followup_interval_days`
- **appointments**: `id`, `clinic_id`, `patient_id`, `practitioner_id`, `service_id`, `start_at`, `end_at`, `status`, `notes`
- **visits** (الزيارة الفعلية/الخدمة المنفّذة): `id`, `clinic_id`, `patient_id`, `appointment_id`, `practitioner_id`, `service_id`, `date`, `medical_notes`
- **sales** (الفاتورة): `id`, `clinic_id`, `patient_id`, `date`, `total`, `payment_method`, `status`
- **sale_items**: `id`, `sale_id`, `service_id`, `price`, `quantity`, `practitioner_id`
- **patient_packages** (عدّاد جلسات الباقة): `id`, `clinic_id`, `patient_id`, `service_id`, `total_sessions`, `used_sessions`, `purchased_at`
- **message_log** (سجل التواصل): `id`, `clinic_id`, `patient_id`, `type`, `sent_at`, `sent_by`, `channel`
- **followups** (المتابعات المستحقة): `id`, `clinic_id`, `patient_id`, `due_date`, `reason`, `status`

**العلاقات:** عيادة → لديها مستخدمين وعملاء وخدمات. عميل → لديه مواعيد وزيارات ومبيعات وباقات ومتابعات. الموعد → يتحوّل إلى زيارة عند اكتماله.

## ٧. معمارية تعدد المستأجرين (Multi-Tenancy)

**النهج المختار:** قاعدة بيانات واحدة مشتركة + عمود `clinic_id` على كل صف + **عزل على مستوى الصفوف (Row Level Security)**. هذا هو النهج العملي والأوفر تكلفة لمنتج SaaS في مراحله المبكرة.

**كيف يعمل العزل:** كل مستخدم مرتبط بعيادة واحدة. عند تسجيل الدخول، هوية العيادة (`clinic_id`) تُحمَّل في سياق المستخدم، وسياسات قاعدة البيانات تمنع أي مستخدم من رؤية أو تعديل بيانات عيادة أخرى — **حتى لو حصل خطأ برمجي في الواجهة**. هذا الأمان على مستوى قاعدة البيانات ضروري ولا يُتنازل عنه في نظام يحمل بيانات طبية.

## ٨. الحزمة التقنية المقترحة (Tech Stack)

اخترتها بناءً على معيارين: **صديقة لـ Claude Code** (موثّقة جيدًا وشائعة) و**قابلة للإدارة من مطوّر واحد**.

| الطبقة | الأداة المقترحة | السبب |
|---|---|---|
| الواجهة | **Next.js (App Router) + TypeScript** | إطار حديث، ممتاز لـ SaaS، Claude Code يتعامل معه بكفاءة عالية |
| التصميم | **Tailwind CSS** + مكتبة مكوّنات (مثل shadcn/ui) | RTL-friendly عبر الخصائص المنطقية (`ms-`, `me-`)، سريع للبناء |
| قاعدة البيانات + المصادقة + التخزين | **Supabase (Postgres)** | يوفّر: Postgres + مصادقة بالأدوار + **RLS جاهز للـ multi-tenancy** + تخزين للصور لاحقًا |
| الترجمة | **next-intl** | إدارة العربية/الإنجليزية وتبديل الاتجاه |
| PWA | إعداد PWA على Next.js | يعمل كتطبيق على الموبايل والتابلت بدون متاجر تطبيقات |
| الرسوم البيانية | **Recharts** | تقارير بصرية بسيطة |

> **بديل عند التوسّع:** لو احتجنا تحكّمًا أعمق في الاستضافة لأسباب قانونية (انظر القسم ٩)، يمكن لاحقًا استضافة Supabase ذاتيًا (self-host) على خادم في المنطقة.

### نقطة حرجة — مكان استضافة البيانات (Data Residency)
قانون البيانات الأردني **يقيّد نقل البيانات الشخصية خارج الأردن** إلا بشروط (منها موافقة صريحة). Supabase المُدار يتيح اختيار المنطقة، لكن أقرب منطقة غالبًا في أوروبا وليست داخل الأردن. الخيارات:
**القرار المعتمد ✅:** نبدأ بـ **Supabase مُدار** — أقرب منطقة مناسبة (غالبًا أوروبا) + **بند موافقة صريح** في نموذج تسجيل العميل يوضّح أن البيانات تُعالَج خارج الأردن. هذا المسار الأسرع للانطلاق، ونؤجّل الاستضافة داخل الأردن لمرحلة التوسّع (حيث يمكن استخدامها كنقطة بيع للعيادات: "بياناتك داخل الأردن").

> **إجراء مطلوب:** صياغة نص الموافقة على المعالجة خارج الأردن يجب أن تكون واضحة وصريحة، ويُفضّل مراجعتها قانونيًا قبل الإطلاق التجاري.

## ٩. الأمان والامتثال — قانون حماية البيانات الأردني رقم ٢٤ لسنة ٢٠٢٣

القانون نافذ منذ ١٧ آذار ٢٠٢٤، ويصنّف **البيانات الصحية كـ"بيانات شخصية حساسة"**. أهم ما يعنينا هندسيًا:

- **الموافقة الصريحة والخطية:** يجب الحصول على موافقة واضحة ومكتوبة قبل معالجة بيانات العميل، محدّدة الغرض والمدّة. → نبني **نموذج موافقة رقمي** عند تسجيل كل عميل جديد، مع تسجيل وقت الموافقة ونسختها.
- **تقييد النقل خارج الأردن** (انظر القسم ٨).
- **الإخطار بالاختراق:** التزام بإخطار أصحاب البيانات عند أي خرق أمني. → نحتاج **سجل تدقيق (audit log)** وخطة استجابة.
- **أمن البيانات:** تشفير أثناء النقل والتخزين، وضوابط وصول صارمة. → تشفير + RLS + حصر الملاحظات الطبية على الأدوار المخوّلة.
- **حقوق صاحب البيانات:** حذف/تصحيح البيانات عند الطلب. → نصمّم القدرة على حذف/تصدير بيانات عميل بعينه.
- **العقوبات:** غرامات قد تصل إلى نسبة من الإيراد السنوي — سبب كافٍ لأخذ الامتثال بجدية من البداية.

> **تنبيه:** أنا لست جهة قانونية، والوثيقة هنا تلخّص المتطلبات العامة لتوجيه التصميم فقط. قبل الإطلاق التجاري يُفضّل مراجعة محامٍ متخصص بحماية البيانات في الأردن — خصوصًا لصياغة نص الموافقة وبند معالجة البيانات في عقد الاشتراك مع العيادات.

## ١٠. الترجمة ودعم الاتجاهين (i18n + RTL/LTR)

- كل النصوص في ملفات ترجمة (عربي/إنجليزي)، لا نصوص مكتوبة داخل الكود مباشرة.
- التبديل بين اللغتين يغيّر اتجاه الصفحة (`dir="rtl"` / `dir="ltr"`) تلقائيًا.
- استخدام **الخصائص المنطقية** في التصميم (`start`/`end` بدل `left`/`right`) حتى ينعكس التخطيط تلقائيًا.
- الأرقام والتواريخ والعملة (دينار أردني) تُنسَّق حسب اللغة.
- اختيار خط عربي واضح للواجهة.

## ١١. خارطة الطريق المرحلية

**المرحلة ٠ — الأساس (قبل أي ميزة):**
إعداد المشروع، المصادقة، الأدوار، معمارية multi-tenant + RLS، الهيكل ثنائي اللغة. *لا نكتب أي ميزة قبل أن يكون عزل المستأجرين مضمونًا ومختبَرًا.*

**المرحلة ١ — الـ MVP:**
العملاء → الخدمات → المواعيد → المبيعات → إعادة الاستهداف اليدوية → التقارير الأساسية. + نموذج الموافقة الرقمي.

**المرحلة ٢ — الأتمتة والتوسّع:**
WhatsApp API الأوتوماتيكي، صور قبل/بعد مع الموافقة، المخزون، العمولات.

**المرحلة ٣ — النضج التجاري:**
الحجز الإلكتروني للعملاء، الدفع الإلكتروني، الفوترة الآلية للاشتراكات، لوحة تحليلات متقدمة.

## ١٢. أسلوب العمل مع Claude Code

- **ابدأ من المرحلة ٠ ولا تتجاوزها.** عزل المستأجرين هو حجر الأساس؛ أي خطأ فيه لاحقًا مكلف جدًا.
- **افتح مشروعًا وأعطِ Claude Code هذه الوثيقة كسياق** (context) في بداية كل جلسة عمل، ووجّهه لمرحلة/وحدة واحدة في كل مرة.
- **اطلب اختبارات لعزل المستأجرين تحديدًا:** تأكيد أن مستخدم العيادة (أ) لا يستطيع بأي طريقة رؤية بيانات العيادة (ب).
- **ابنِ وحدة كاملة قبل الانتقال للتالية** (العملاء أولًا كاملة، ثم الخدمات، ثم المواعيد...).
- **اعمل commit صغيرة ومتكررة** حتى يسهل التراجع.
